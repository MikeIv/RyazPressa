# Вариант 3 — Реестр изменений для DevOps (Backend + Инфраструктура)

**Цель варианта:**  
Фронт собирается и отдаётся как чистая SPA-статика. Вся логика определения сайта, конфиг и данные переносятся на бэкенд. Дополнительного Node/Nitro/edge-процесса на сервере не требуется.

**Количество сайтов:** 26 (25 базовых изданий + ryazpressa).

---

## 1. Инфраструктура / DevOps

| Что нужно                                                                                                                      | Примечание                                                                                                                                           |
| ------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Отдавать фронт как чистую статику SPA, без серверной части                                                                     | Сейчас приложение использует Nitro даже при `ssr: false`. Nitro выполняет роль BFF + определения сайта. В этом варианте Nitro полностью убирается.   |
| После сборки (`pnpm build`) использовать только каталог со статикой (`.output/public` или эквивалент после статического билда) | Артефакты: `index.html`, `_nuxt/*` (хэшированные JS/CSS), `/sites/*` (логотипы и ассеты), favicon, apple-touch-icon. Серверная часть Nitro не нужна. |
| Настроить SPA fallback: все пути, кроме `/api/*`, должны возвращать `index.html`                                               | Реальные файлы (`_nuxt/*`, `/sites/*` и др.) отдаются как обычно. Это стандартная настройка для SPA.                                                 |
| Кэширование: `_nuxt/*` — `public, max-age=31536000, immutable`                                                                 | Длинный кэш для хэшированных ассетов. Остальные ресурсы (`/sites/*`, favicon, apple-touch-icon) — обычный кэш или с валидацией.                      |
| Все контент-домены (apex + www + punycode/IDN) отдают один и тот же набор статических файлов + один и тот же API-слой          | Примерно 26 доменов. Полный список доменов из `shared/sites/manifest.json` + ryazpressa можем предоставить.                                          |
| Не требуется поднимать дополнительный Node.js / Nitro / edge-процесс на сервере                                                | Бэкенд отдаёт статику SPA вместе со своим API на тех же доменах, где сейчас живут сайты. Это ключевое требование для снижения нагрузки.              |

## 2. API-эндпоинты (что должен отдавать бэкенд)

SPA будет продолжать вызывать относительные пути `/api/*` на тех же доменах.

**Общие требования:**

- Пагинация: `{ "data": [...], "meta": { "page": 1, "perPage": 10, "total": 42, "totalPages": 5 } }`
- Ошибки: `{ "statusCode": 404, "message": "..." }`
- Если раздел выключен в `sections` сайта — реальный **404**

| Эндпоинт                | Метод | Когда доступен                                 | Примечание                                                            |
| ----------------------- | ----- | ---------------------------------------------- | --------------------------------------------------------------------- |
| `/api/_site`            | GET   | всегда                                         | Критично. См. раздел 3                                                |
| `/api/news`             | GET   | всегда                                         | `page`, `perPage`, `category?`, `district?`, `period?` (см. раздел 4) |
| `/api/news/{slug}`      | GET   | всегда                                         | Полная статья                                                         |
| `/api/gallery`          | GET   | `sections.gallery === true`                    | Иначе 404                                                             |
| `/api/documents`        | GET   | `sections.documents === true`                  | Иначе 404                                                             |
| `/api/contacts`         | GET   | `sections.contacts === true`                   | Иначе 404, объект без пагинации                                       |
| `/api/districts`        | GET   | `sections.okruga === true` (только ryazpressa) | Иначе 404, формат `{ "data": [...] }`                                 |
| `/api/districts/{slug}` | GET   | `sections.okruga === true`                     | `DistrictDetail` (округ + массив новостей)                            |
| `/api/projects`         | GET   | `sections.projects === true`                   | Иначе 404, `{ "data": [...] }`                                        |
| `/api/projects/{slug}`  | GET   | `sections.projects === true`                   | `ProjectDetail`                                                       |
| `/api/near-us`          | GET   | `sections.ryadomSNami === true`                | Иначе 404, статичный `{ "title", "content" }`                         |

## 3. Конфиг сайта — `GET /api/_site`

**Критически важный эндпоинт.** SPA вызывает его постоянно (страницы, middleware, composables).

Бэкенд должен по `Host` (или `X-Forwarded-Host`) определять сайт и возвращать `PublicSiteConfig`.

Обязательные поля:

- `slug`, `name`
- `apiSiteHost` (основной домен без www)
- `articlePathPrefix` (`""` или `"/news"`) — сильно влияет на роутинг статей
- `sections` (6 булевых флагов)
- `nav` (уже посчитанный массив)
- `theme` (цвета, радиусы, логотип, размеры, favicon, headerGradientStart)

**Важно:** значения `sections`, `articlePathPrefix`, `theme` и `nav` для каждого из 26 сайтов должны быть актуальными и синхронизированными с фронтом.

## 4. Новости — специальная обработка

Сейчас эту логику выполняет Nitro-прокси. Её нужно перенести на бэкенд (или согласовать упрощение).

Что происходит сейчас:

- Путь `/api/news` маппится на `/api/posts` бэкенда.
- Параметр `period=today-yesterday` удаляется перед вызовом.
- Ответ нормализуется (Laravel-формат → контракт фронта).
- При `period=today-yesterday` фронт дополнительно фильтрует новости по датам (сегодня/вчера) и переписывает `meta.total` / `meta.totalPages`.

Где используется: главная страница (лента + блок «Главное сегодня»).

**Вопрос на согласование:** бэкенд реализует эту логику, или фронт будет работать с сырыми данными `/api/posts`?

## 5. Контроль доступности разделов (section gating)

Если для текущего сайта `sections.xxx === false`, то эндпоинт этого раздела должен отдавать **404**.

Это требование текущего контракта (`docs/api-contract.md`). Нужно сохранить поведение.

## 6. Реестр сайтов и синхронизация конфигов

Сейчас реестр и конфиги живут во фронте:

- `shared/sites/manifest.json` (25 базовых изданий)
- `shared/sites/ryazpressa.ts` (расширенный сайт)
- `shared/sites/baseSections.ts`, `createBaseSite.ts`, `siteBrandColors.ts` и др.

В Варианте 3 бэкенду нужно будет:

- Либо стать владельцем этого реестра (или его части),
- Либо получать актуальные данные из фронтового репозитория.

Особенно важно синхронизировать `sections`, `articlePathPrefix`, `theme` и список доменов.

## 7. Сборка, деплой, локальная разработка и тесты

- Сборка фронта должна производить только статику (без Nitro).
- DevOps/инфраструктура принимает артефакты фронта и отдаёт их вместе с API на одних и тех же доменах.
- Локальная разработка фронта: текущая схема (`pnpm dev` + mock + `NUXT_USE_MOCK_API`) завязана на Nitro. Нужно будет адаптировать (временный мок, отдельный dev-сервер и т.д.).
- Smoke-тесты (`pnpm smoke:sites`) и эмуляция через `Host`-заголовок требуют переработки.
